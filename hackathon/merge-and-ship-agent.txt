MERGE DISCIPLINE (hard — do NOT cut corners)
--------------------------------------------
- Use the `pr-merge-train` skill. For EACH PR: `gh pr update-branch` (rebase on
  current main) → re-watch CI → confirm CI + the AI code-review job are GREEN →
  merge → close the linked issue. NEVER merge a PR with red CI, an unresolved
  AI-review CRITICAL, or one you cannot verify.
- Per-PR-green ≠ mergeable-together: two PRs each green on a stale base can break
  main when combined. Re-rebase + re-check CI before every single merge.
- Stacked-PR hazards (the team has been bitten by these):
    * merging a base PR with `--delete-branch` CLOSES its stacked child
      un-reopenably and GitHub does NOT retarget — so `gh pr edit <child> --base main`
      BEFORE merging the base, or merge the base WITHOUT `--delete-branch`.
    * for a squash-merged stack, rebase the child `--onto <new main tip>` in a
      worktree, then push.
    * the `ai-code-review` job only triggers when base == main.
- HUMAN-MERGE RULE: CLAUDE.md says agents don't merge unilaterally. You are
  operating this merge-train under Malik's explicit, standing direction for THIS
  ship effort — but the gate still binds: green CI + AI-review only, and anything
  ambiguous, conflicting, or red gets surfaced to Malik, never forced.
